# ADR-40030: Auto-detect GHES host during compile from the git origin remote

**Date**: 2026-06-18
**Status**: Draft

## Context

`gh aw compile` performs `gh`-backed checks (notably `--validate`, which queries repository features) that must target the correct GitHub host. The `add` and `audit` commands already infer the host from the git origin remote, but `compile` did not. As a result, when `GH_HOST` was unset and the repository lived on a GitHub Enterprise Server (GHES) host, compile sent feature checks to `github.com` instead of the GHES host, and downstream host-pinning logic never engaged. This left compile behavior inconsistent with the rest of the CLI and silently incorrect in GHES environments.

## Decision

We will add the existing git-remote-based host-detection pattern to `CompileWorkflows`. When `GH_HOST` is unset and the origin remote resolves to a non-`github.com`, non-empty host, compile calls `workflow.SetDefaultGHHost(detectedHost)` before validation and other `gh`-backed checks run. An explicitly set `GH_HOST` is never overridden, and `github.com` remotes are left untouched. This makes compile's host resolution consistent with `add` and `audit`.

## Alternatives Considered

### Alternative 1: Require users to set `GH_HOST` explicitly on GHES
Document that GHES users must export `GH_HOST` before running compile. Rejected because it places an easily-forgotten burden on users, produces silent misbehavior when omitted, and is inconsistent with `add`/`audit`, which already auto-detect.

### Alternative 2: Extract host detection into a single shared entry point invoked by all commands
Centralize detection in one place (e.g., a shared pre-run hook) so every command inherits it. Rejected for this PR's scope: it is a larger refactor touching multiple command paths, while the immediate need is to bring compile to parity using the already-proven `getHostFromOriginRemote()` helper. The shared approach remains a reasonable future follow-up.

## Consequences

### Positive
- Compile `--validate` and other `gh`-backed checks target the correct GHES host, fixing silently-wrong validation.
- Compile host resolution is now consistent with `add` and `audit`.

### Negative
- Introduces implicit mutation of global host state (`SetDefaultGHHost`) as a side effect of compile startup, which can surprise readers tracing host selection.
- Couples compile correctness to the git origin remote; an unexpected or misconfigured remote could redirect checks to the wrong host.

### Neutral
- Behavior is unchanged for `github.com` remotes and when `GH_HOST` is explicitly set.
- Regression coverage added in `pkg/cli/compile_orchestrator_host_test.go` for the GHES, explicit-`GH_HOST`, and `github.com` cases.
