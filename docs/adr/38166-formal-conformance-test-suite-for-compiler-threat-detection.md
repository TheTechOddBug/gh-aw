# ADR-38166: Formal Conformance Test Suite for Compiler Threat Detection

**Date**: 2026-06-09
**Status**: Draft

## Context

The compiler-side threat controls (CTR-001, CTR-011, CTR-014, CTR-015, CTR-016) are specified normatively in `specs/compiler-threat-detection-spec.md` (v1.0.13), but the existing tests for these controls were scattered and not traceable back to individual spec predicates. There was no convention that tied each control's invariants to a dedicated, machine-checkable test, nor a guarantee that such tests exercised the real compiler logic rather than reimplemented or stubbed approximations. As the threat-detection surface grows, the project needs conformance tests that stay anchored to production behavior and map clearly to the spec so that regressions in security-critical validators are caught deterministically.

## Decision

We will adopt a formal conformance test suite convention for compiler threat detection: tests are grouped under a `TestFormal_<CTR-ID>_<Predicate>` naming scheme, one file (`pkg/workflow/compiler_threat_detection_formal_test.go`) mapping spec predicates and edge cases to assertions. Each test **calls the production validator entry points directly** (`EnforceSafeUpdate`, `validateDangerousPermissions`, `validateNetworkFirewallConfig`, `validateStrictNetwork`, `validateSafeOutputsAllowedLabelsGlobScope`, `validateRunInstallScripts`) rather than stubbing or mocking them, so conformance assertions remain anchored to real compiler behavior and stay in sync with the spec.

## Alternatives Considered

### Alternative 1: Stub or mock the validators in tests
Replace the production validators with test doubles to isolate the test logic from internal implementation details. Rejected because doubles drift from real behavior over time, producing tests that pass while the actual compiler diverges from the spec — defeating the purpose of conformance testing for security-critical controls.

### Alternative 2: Keep informal, scattered unit tests
Continue adding ad-hoc unit tests near each validator without a unifying naming convention or spec mapping. Rejected because it provides no traceability from spec predicate to test, making it hard to confirm complete coverage of each CTR control or to spot missing edge cases.

### Alternative 3: External formal-verification tooling
Use a separate formal-methods/model-checking tool to verify the spec predicates against the implementation. Rejected as disproportionate for the current scope: it adds heavy tooling and a second source of truth, whereas Go table-style tests against live entry points give adequate, low-friction conformance coverage.

## Consequences

### Positive
- Each CTR predicate and edge case maps to a named test, giving clear spec-to-test traceability and coverage visibility.
- Tests run against production validators, so spec violations introduced by future refactors are caught deterministically rather than masked by stubs.

### Negative
- Tests are coupled to internal validator signatures (e.g., `validateDangerousPermissions`, `validateStrictNetwork`), so refactoring those signatures requires updating the suite.
- Asserting on substrings of error messages (e.g., `"allow-urls requires ssl-bump: true"`, `"CTR-015"`) makes tests sensitive to error-message wording changes.

### Neutral
- The suite is built with the `!integration` build tag and depends on `testify` (`assert`/`require`), consistent with the existing `pkg/workflow` test conventions.
- The `TestFormal_` naming convention becomes the expected pattern for future compiler threat-detection conformance tests.
